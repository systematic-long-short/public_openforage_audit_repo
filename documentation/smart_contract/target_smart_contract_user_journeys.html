What you most need to know — in 60 seconds

  • The headline. Of 77 on-chain journeys the protocol needs, the implemented target now fully covers 75, with 1 accepted PARTIAL (X-2, off-chain trading auditability) and 1 deliberate NONE (K-7, the removed per-instruction trading gate). Every design gap was solved on 2026-06-08, and the former build gaps were closed in the June 2026 target-gap buildout: USDCTreasury, FORAGETreasury, the slimmed HLTradingBridge, the 4-of-7 guardian machinery, the successor registry, and directional freeze are now implemented and proven on Arbitrum Sepolia.
  • Two separate axes, never conflated. Design coverage asks "does the target design specify a path?" (FULL / PARTIAL / NONE). Build status asks "does the code exist?" (live / implemented · off-chain · gap). The verdict scores the design; implementation evidence is shown separately in the build-status column.
  • The Library has no on-chain journey. Contributors register, submit signals, and are scored entirely off-chain. The on-chain boundary is the protocol's own server crediting an agent's address — the contributor never signs an on-chain transaction. The target SC design contains no "contributor / attribution" concept (§11).
  • Three facts shape every row. (1) The target consolidations and guardian extensions are now committed code and deployed on Arbitrum Sepolia. (2) The Library is off-chain. (3) The target deliberately diverged from the historical deployed-code baseline on economics and safety (§15), and that divergence is now the implemented target.
§1

How to read this map

Two axes, six columns, one citation per row.

Design coverage — does the target design specify a path?

FULL  the design fully specifies the journey.

PARTIAL  intent exists, but an API, eligibility, or control path is missing, inconsistent, or under-specified.

NONE  no on-chain journey in the target design (by omission or by deliberate removal).

Build status — does the code exist?

🟢 live  exists in committed Solidity and is part of the target implementation.

🟢 implemented  was formerly target-only and now exists in committed Solidity.

⚪ off-chain  not a smart-contract journey by design.

🔴 gap  the design names the need but specifies no contract/API.

The six columns are: Journey · Design coverage · Build status · Fulfilled by (the contract/function) · Gap / owner decision · Evidence. Evidence tagged TSC:<line> is the target architecture doc; (code) is committed Solidity in openforage_smart_contracts/src/; (ext) is an off-chain source (the Library or the signal service). A dual build chip (e.g. 🟢/⚪) means the on-chain settlement surface exists but part of the journey remains deliberately off-chain.

What was resolved, and what remains

The 16 PARTIAL + 2 NONE rows, driven to closure on 2026-06-08 — followed by the 2026-06-09 target implementation and Sepolia proof.

Design gaps — all closed. Each former gap was resolved by one of four moves; none is an open design question any more:

  1. Guardian defence-in-depth folded into the target (E-2, E-6, E-7, G-9, G-10, K-8). The 4-of-7 Tier-2 quorum, the named preCommittedSuccessors[role] registry on the GuardianModule, and the directional (asymmetric) freeze were specified in §6 and §13, ported from the protocol's defence-in-depth work, and are now implemented. TSC:824-927, 2050-2102
  2. Depositor mechanics reconciled to the deployed code (D-3, D-9, D-10b, D-12). The FORAGE priority-lane escrow lifecycle, auto-renew / Tier-0-reversion at lock-up maturity (D-9 was mislabelled NONE), the tier-share transfer rules, and the large-position model (hard-revert + queue waitlist) are now documented from the code (P38). 🟢 live. TSC:1617-1655, 1741-1748, 1924-1961
  3. Custody & bridge given an honest design (K-4, K-5, K-10, K-13, K-12). The withdrawal-intent control is stated honestly — the chain cannot enforce the off-chain withdraw3 signature, so it is keeper-policy + an on-chain intent record + arrival reconciliation + monitoring (residual in X-4). Two-step executor/keeper wiring, the Bridge2 pin, the registry↔bridge cap reconciliation (10%/10%), and a Bridge2-only bridge (no CCTP migration planned) are all settled. TSC:2227-2347
  4. Agent eligibility kept off-chain by owner decision (T-3, L-4, L-5, L-6). On-chain disbursement is fully designed; who-earned-what is computed off-chain (Merkle roots / disburse list) — a deliberate, accepted boundary. TSC:1780-1807

What remains is production readiness, not target buildout. The three target contracts — USDCTreasury, FORAGETreasury, and the slimmed HLTradingBridge — plus the 4-of-7 guardian machinery and the successor registry are now committed code. Arbitrum Sepolia A25 re-proved the flow and ended with the deployment post-tail frozen; external audit sign-off and mainnet deployment are the remaining maturity steps. TSC:383-398, 1432-1456

Two accepted non-goals, not gaps. The on-chain per-instruction trading gate (K-7, NONE) stays deliberately removed for the off-chain-custody pivot, and full on-chain auditability of trading (X-2, PARTIAL) is necessarily off-chain — telemetry (P34) is the audit trail. Both are owner-ratified (2026-06-08); the residual that the cold account is a trusted custody root is accepted in X-4.
